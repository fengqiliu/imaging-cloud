## 1. 概述

### 1.1 背景与目标

医疗影像PACS（Picture Archiving and Communication System）系统是医院信息化的核心基础设施，承载影像采集、存储、分发、诊断和归档的全生命周期管理。随着AI技术和分布式系统的发展，传统单体PACS架构在以下方面面临挑战：

| 挑战维度 | 传统PACS痛点 | Agent架构解法 |
|---------|------------|--------------|
| 处理能力 | 峰值时段影像积压严重 | Agent弹性扩缩容，按需调度 |
| 智能化 | 人工阅片效率低，漏诊风险高 | AI诊断Agent，多模型融合辅助 |
| 可靠性 | 单点故障影响全院业务 | Agent故障隔离，自动故障转移 |
| 扩展性 | 新功能上线需要停机发布 | Agent独立部署，热更新 |
| 合规性 | 操作行为难以追溯审计 | 安全审计Agent，全链路追踪 |

### 1.2 设计范围

本文档覆盖以下设计内容：

- **业务场景**：8大核心业务域的Agent化拆解
- **数据场景**：6类医疗数据的处理策略
- **功能模块**：9个专职Agent的详细设计
- **架构模式**：事件驱动编排 + 层次化多Agent两种核心模式
- **落地路线**：4阶段渐进式实施计划

### 1.3 术语定义

| 术语 | 说明 |
|-----|------|
| Agent | 具备自主感知、决策、行动能力的自治服务单元 |
| Orchestration Agent | 负责协调其他Agent的主控Agent |
| Event | Agent间传递的异步消息，携带业务语义 |
| Tool | Agent可调用的能力接口（API、数据库、文件系统等） |
| SLA | 服务级别协议，定义响应时间和可用性要求 |
| DICOM | 医学数字成像和通信标准 |
| WADO | 影像对象的网络访问协议 |

## 3. 数据场景分析

### 3.1 医疗数据类型与处理策略

| 数据维度 | 特征描述 | 规模估算 | Agent处理策略 |
|---------|---------|---------|--------------|
| **DICOM影像文件** | 单文件1-50MB，每次检查50-2000张，格式标准但版本多 | 胸部CT单次≈500MB | 流式处理 + 并发解析，批量入库 + 元数据索引 |
| **患者主数据** | 身份+历史检查+用药史，跨院区去重合并 | 百万级患者记录 | MDM主数据管理，实体匹配+融合去重 |
| **影像元数据** | DICOM标签约300个字段，多模态CT/MRI/DR/US/PET | 每张影像≈50KB元数据 | 选择性提取+全文索引，模态感知路由 |
| **报告文本** | 结构化模板+自由文本，ICD/SNOMED编码关联 | 每份报告≈2KB | NLP解析+术语标准化，知识图谱抽取 |
| **AI推理数据** | 像素级标注+概率分布，多模型集成+不确定度量化 | GPU推理≈10-30秒/次 | GPU调度+结果融合，模型路由+置信度聚合 |
| **操作日志** | 高频写入>10万条/天，安全审计+访问轨迹 | >50GB/年 | 异步写入+流式聚合，实时分析+异常检测 |

### 3.2 数据分级存储模型

```
  访问热度
     ▲
     │  🔥 HOT TIER (本地SSD / MinIO)
     │  ├── 范围：近30天 + 活跃中检查
     │  ├── 存储：NVMe SSD，RAID10
     │  ├── 访问延迟：< 100ms
     │  └── 容量：约占总量 15%
     │
     │  🌡️ WARM TIER (对象存储 / NAS)
     │  ├── 范围：30天 - 1年，定期压缩
     │  ├── 存储：SATA HDD，RAID6
     │  ├── 访问延迟：< 1s
     │  └── 容量：约占总量 35%
     │
     │  🧊 COLD TIER (归档存储 / 磁带库)
     │  ├── 范围：1年以上，高压缩比，合规保留
     │  ├── 存储：磁带/冷对象存储
     │  ├── 访问延迟：< 5min（需预热）
     │  └── 容量：约占总量 50%
     │
     └──────────────────────────────▶ 时间轴
        0    30天    1年    5年   15年（法规要求）
```

### 4.2 Agent-01: DICOM接收Agent (DicomIngestAgent)

**职责定义**: 负责DICOM网络协议接收、初步验证及消息入队，是整个影像处理流水线的入口。

**触发条件**:
- C-STORE SCP（DICOM推送，标准端口11112）
- STOW-RS HTTP接口（RESTful方式上传）
- 本地目录监听（文件拖入批量处理）

**处理流程**:

```
  接收DICOM文件
       │
       ▼
  校验SOPClassUID合法性 ──失败──▶ 拒绝并返回错误状态
       │ 通过
       ▼
  提取关键标签（患者/检查/序列/影像）
       │
       ▼
  重复检测（Redis缓存SOPInstanceUID，TTL=24h）
       │ 新影像                    已存在
       ▼                              │
  病患匹配/创建                   返回已存在响应
       │
       ▼
  存储DICOM文件（HOT层）
       │
       ▼
  写入cf_image记录（status=received）
       │
       ▼
  发布DicomReceivedEvent到MQ
       │
       ▼
  返回C-STORE成功ACK
```

**工具集**:

| 工具 | 用途 |
|-----|------|
| dcm4che-net | DICOM SCP服务实现 |
| DicomParser | DICOM标签提取 |
| MQ生产者 | 事件发布 |
| Redis | 去重缓存（SOPInstanceUID） |
| StorageService | 文件持久化 |

**输出**:
- 事件：`DicomReceivedEvent { examId, imageId, filePath, modality, studyInstanceUID }`
- 状态：`cf_image.status = 'received'`

**关键指标**:
- 接收吞吐量：≥ 100张/秒（峰值）
- 单文件处理延迟：< 500ms
- 去重准确率：100%（基于SOPInstanceUID）

### 4.3 Agent-02: 影像质量控制Agent (ImageQCAgent)

**职责定义**: 对接收到的DICOM影像进行自动化质量评估，标记问题影像，触发重照通知。

**订阅事件**: `DicomReceivedEvent`

**评估维度矩阵**:

| 评估类别 | 检测项目 | 权重 | 不通过处理 |
|---------|---------|------|---------|
| **技术质量** | 曝光度异常/噪声超标/图像模糊/运动伪影 | 40% | 标记重照 |
| **完整性** | 必要DICOM标签缺失/序列帧数不完整 | 30% | 补充采集 |
| **一致性** | 患者信息与HIS不一致/影像方向错误 | 20% | 人工核查 |
| **合规性** | 个人信息未脱敏/无有效授权 | 10% | 阻断流程 |

**评分规则**:
```
综合评分 = Σ(各维度评分 × 权重)

评级：
  90-100分：优秀，直接进入AI分析
  70-89分：合格，附质量提示进入流程
  50-69分：勘用，需医生确认后继续
  0-49分：不合格，触发重照流程
```

**动作链**:
```
  QC评分
    │
    ├── score ≥ 70 ──▶ 发布 ImageQCPassedEvent ──▶ 继续AI分析
    │
    └── score < 70 ──▶ 创建QC问题记录
                        │
                        ├── 通知放射技师（站内消息 + 短信）
                        ├── 挂起检查状态（exam_status = 'qc_failed'）
                        └── 记录问题影像到 cf_qc_issue 表
```

**输出**:
- 字段：`cf_image.qc_score`, `cf_image.qc_status`, `cf_image.qc_issues`
- 事件：`ImageQCPassedEvent` 或 `ImageQCFailedEvent`

### 4.4 Agent-03: 存储调度Agent (StorageSchedulerAgent)

**职责定义**: 执行分级存储迁移策略，管理存储容量，实现影像数据生命周期自动化管理。

**存储分级策略**:

```
  分级  存储介质         触发迁移条件              访问延迟  压缩策略
  ─────────────────────────────────────────────────────────────────────
  HOT   本地SSD/MinIO    入库即存入                < 100ms  无压缩
  WARM  对象存储/NAS     30天未访问 OR 检查已完成   < 1s     无损压缩(JPEG2000)
  COLD  归档存储         1年未访问 OR 法规归档       < 5min  高压缩(有损可选)
```

**调度触发机制**:

| 触发类型 | 条件 | 执行动作 |
|---------|------|---------|
| 定时扫描 | 每日 02:00 Cron | 批量扫描迁移候选影像 |
| 存储水位告警 | HOT层 > 80% | 紧急迁移最旧的20%影像 |
| 主动访问预热 | COLD层影像被请求 | 后台异步预热到HOT层 |
| 容量预测 | 预测7天后超阈值 | 提前触发迁移 |

**容量规划模型**:
```
  当前使用量 + 日均增长量 × 预测天数 = 预测容量

  告警阈值:
  ├── 警告(Warning): 使用率 > 70%
  ├── 严重(Critical): 使用率 > 85%
  └── 紧急(Emergency): 使用率 > 95% → 阻断新影像写入
```

### 4.5 Agent-04: AI诊断辅助Agent (AIDiagnosisAgent)

**职责定义**: 编排调度多个AI模型，融合分析结果，为医生提供高置信度的诊断辅助建议。

**模型路由策略（按影像模态）**:

| 模态 | 调度模型 | 分析能力 | 响应时间目标 |
|-----|---------|---------|------------|
| CT | 肺结节检测 + 骨折识别 + 颅内出血 | 病灶定位、体积测量 | < 30s |
| MRI | 脑肿瘤分割 + 脊柱分析 + 软组织识别 | 语义分割、区域标注 | < 60s |
| DR/CR | 胸片分析 + 骨密度评估 + 异物检测 | 分类识别、区域标注 | < 10s |
| 超声 | 器官测量 + 甲状腺结节 + 胎儿评估 | 测量辅助、风险分级 | < 15s |
| PET-CT | 代谢活性分析 + 肿瘤分期 + 疗效评估 | 多模态融合分析 | < 120s |

**多模型结果融合（Ensemble）**:

```
  模型1输出 ──┐
  模型2输出 ──┼──▶ 加权投票 ──▶ 不确定度量化 ──▶ 置信区间计算
  模型3输出 ──┘         │
                        ▼
                   最终诊断建议
                        │
              ┌─────────┼─────────┐
              ▼         ▼         ▼
           高置信度   中置信度   低置信度
           (>85%)    (60-85%)   (<60%)
              │         │         │
           直接展示   高亮提示   标注不确定
```

**人机协作接口**:
```
  AI结果展示:
  ├── 关键发现高亮标注（影像叠加层）
  ├── 置信度百分比展示
  ├── 相关文献参考
  └── 医生操作: 接受 / 修改 / 拒绝

  反馈闭环:
  医生反馈 ──▶ 标注数据集 ──▶ 模型迭代训练 ──▶ 精度持续提升
```

**写入表**: `cf_ai_interpret`（现有数据库表，代码层待实现）

### 4.6 Agent-05: 报告生成Agent (ReportGenerationAgent)

**职责定义**: 基于AI解读结果和历史数据，智能匹配报告模板，辅助医生生成结构化诊断报告，并渲染为标准PDF。

**触发链**:
```
  检查完成事件 ──┐
                 ├──▶ 等待报告状态 ──▶ 医生打开报告界面 ──▶ 报告生成流程
  AI解读完成  ──┘
```

**报告生成六步流程**:

```
  Step 1: 模板选择
  ├── 输入：影像模态 + 检查部位 + 患者年龄段
  └── 输出：最优结构化报告模板

  Step 2: AI内容预填
  ├── 输入：cf_ai_interpret.findings
  └── 输出：结构化报告段落（检查所见/诊断意见/建议）

  Step 3: 历史对比分析
  ├── 输入：同患者历史检查记录（最近3次）
  └── 输出：病变变化趋势描述

  Step 4: 术语规范化
  ├── 输入：自由文本描述
  └── 输出：标准医学术语 + ICD编码关联

  ├── 输入：模板 + 内容 + 关键影像截图（最多6张）
  └── 输出：符合规范的诊断报告PDF（iText7实现）

  Step 6: 数字签名
  ├── 输入：医生电子签名 + 报告ID
  └── 输出：带数字水印的最终PDF
```

### 4.7 Agent-06: 工作流编排Agent (WorkflowOrchestrationAgent)

### 4.8 Agent-07: 影像分发Agent (ImageDeliveryAgent)

**职责定义**: 为多种客户端提供影像分发服务，实现协议适配、性能优化和带宽管理。

**分发协议适配**:

### 4.9 Agent-08: 安全审计Agent (SecurityAuditAgent)

**职责定义**: 实现全链路访问审计、患者隐私保护、安全合规管理和异常行为实时检测。

**合规框架对齐**:
- 国家卫健委《医疗数据安全管理办法》
- HIPAA（适用于国际化场景）
- DICOM ATNA（审计追踪和节点鉴别）
- GB/T 35273-2020《个人信息安全规范》

**隐私保护机制**:

```
  影像对外分享前自动脱敏:
  ├── DICOM标签清理（PatientName → 脱敏/加密）
  ├── 影像像素层面去标识化（去除烧录文字）
  ├── 患者姓名替换为匿名标识
  └── 联系方式/地址等敏感信息清除

  数据访问最小权限:
  ├── 临床医生：只能访问自己科室患者
  ├── 放射科医生：可访问全院影像，不可修改患者信息
  ├── 患者：只能访问本人影像
  └── 管理员：系统配置，不默认访问临床数据

**异常行为检测规则**:

| 规则ID | 行为描述 | 检测条件 | 处理动作 |
|-------|---------|---------|---------|
| AUD-001 | 批量下载异常 | 单用户 > 50张/分钟 | 实时告警 + 临时限流 |
| AUD-002 | 非工作时间大量访问 | 22:00-06:00访问 > 100次 | 风险评分 + 次日核查 |
| AUD-003 | 权限越界访问 | 访问未授权患者影像 | 实时阻断 + 安全通知 |
| AUD-004 | 异常地理位置 | 境外IP访问内网系统 | 阻断 + 安全团队告警 |
| AUD-005 | 账号异常共享 | 同账号多地同时登录 | 强制下线 + 密码重置提示 |

## 6. Agent间通信协议

### 6.1 通信方式选择矩阵

| 通信场景 | 推荐方式 | 技术实现 | 适用原则 |
|---------|---------|---------|---------|
| 强一致性状态查询 | 同步RPC | HTTP/gRPC | 要求立即返回结果 |
| 高吞吐异步处理 | 消息队列 | Kafka/RabbitMQ | 解耦高并发场景 |
| 多Agent共享数据 | 共享状态 | Redis/数据库 | 多Agent需读同一状态 |
| 实时双向通信 | WebSocket | Spring WebSocket | 实时进度推送 |
| 批量数据传输 | 文件共享 | 共享存储路径 | 大文件DICOM传输 |

### 6.2 标准消息信封格式

```json

### 6.3 消息优先级策略

```
  优先级      适用场景                           队列策略
  ─────────────────────────────────────────────────────────────────
  URGENT      急诊检查、系统告警、安全事件        独立优先队列，立即处理
  HIGH        门诊当日检查、会诊请求              高优先队列，30s内处理
  NORMAL      普通门诊检查、定期归档              标准队列，5min内处理
  LOW         历史数据迁移、统计报告生成           低优先队列，按空闲处理
```

### 6.4 Agent服务发现与注册

```
  启动流程:
  1. Agent启动 → 向注册中心(Nacos/Eureka)注册
     {agentId, agentType, instanceId, host, port, capabilities, status}

  2. 注册中心 → 健康检查(心跳间隔30s, 超时3次摘除)

  3. WorkflowAgent → 查询注册中心 → 获取可用Agent列表

  4. 负载均衡策略:
     ├── RoundRobin: 默认，均匀分配
     ├── WeightedRandom: 根据节点性能权重
     ├── LeastConnections: 连接数最少优先（AI推理Agent）
     └── Consistent Hash: 按examId哈希（保证同一检查在同一节点）
```

### 8.3 稳定性强化设计

#### 8.3.1 消息幂等性实现方案

**问题背景**: Kafka/RabbitMQ在网络抖动或消费者超时时会自动重试，可能导致同一消息被消费多次，造成重复处理。

**幂等消费三层保障**:

```
  第一层：消息去重（生产者）
  ─────────────────────────────────────────────────────
  每个Agent消息必须携带全局唯一 messageId（UUID v4）
  生产者发送前记录已发送（Redis SETNX，TTL=7天）
  发送失败时根据messageId查询是否已发送，避免重复投递

  第二层：消费者幂等（数据库唯一约束）
  ─────────────────────────────────────────────────────
  SQL Server:
    INSERT INTO cf_agent_event (message_id, ...)
    SELECT @messageId, ...
    WHERE NOT EXISTS (
      SELECT 1 FROM cf_agent_event WHERE message_id = @messageId
    );
  -- 影响行数=0时，说明消息已处理，直接ACK

  第三层：业务层幂等（关键操作）
  ─────────────────────────────────────────────────────
  状态变更操作使用乐观锁：
    UPDATE cf_examination
    SET exam_status = @newStatus, version = version + 1
    WHERE id = @id AND version = @currentVersion;
  -- 影响行数=0时，说明状态已被其他操作变更，抛出并发冲突异常
```

**死信队列（Dead Letter Queue）设计**:

```
  正常消费流程:
  Kafka Topic ──▶ [Consumer Group] ──▶ 成功处理 ──▶ ACK偏移量

  失败重试流程:
  失败（可重试）──▶ 重试队列（delay=10s/30s/2min指数退避）──▶ 重试消费
  失败（不可重试）──▶ DLQ Topic（pacs.dicom.received.dlq）──▶ 人工处理 + 告警
  超最大重试（3次）──▶ DLQ + 事件记录cf_agent_event.status='dead_letter'
```

#### 8.3.2 数据库事务边界设计

**文件上传与数据库一致性**:

当前问题：`storageService.upload()`（外部IO）与 `baseMapper.insert()`（数据库）在同一事务中，文件成功上传后数据库失败无法回滚。

**Saga模式解决方案**:

```
  Step 1: 创建数据库记录（status=pending_upload）
  ─────────────────────────────────────────────────────
  INSERT INTO cf_image (..., status) VALUES (..., 'pending_upload');
  ─────────────────────────────────────────────────────
  事务T1：仅包含数据库写，毫秒级，极快提交

  Step 2: 上传文件至存储服务（独立事务外）
  ─────────────────────────────────────────────────────
  storageService.upload(file);  // 文件系统或MinIO
  ─────────────────────────────────────────────────────
  失败处理：
    UPDATE cf_image SET status='upload_failed' WHERE id=@id;
    抛出业务异常，通知重试

  Step 3: 更新状态为received（最终一致性）
  ─────────────────────────────────────────────────────
  UPDATE cf_image SET status='received', file_path=@path WHERE id=@id AND status='pending_upload';
  ─────────────────────────────────────────────────────
  使用条件更新防止重复处理

  Step 4: 发布MQ事件（异步，最终一致性）
  ─────────────────────────────────────────────────────
  MQ发布失败：
    补偿事务：UPDATE cf_image SET status='received_no_event'
    后台任务轮询扫描该状态，重新发布事件（最多重试100次）
```

**分布式锁保护状态机**:

```
  WorkflowOrchestrationAgent 并发冲突场景：
  场景：ImageQCAgent 和 StorageAgent 同时处理同一检查的状态更新

  解决方案：悲观锁（SELECT FOR UPDATE）
  ─────────────────────────────────────────────────────
  BEGIN TRANSACTION;
  SELECT * FROM cf_examination WHERE id = @examId FOR UPDATE;
  -- 锁定检查记录，后续更新串行执行

  检查当前状态是否允许流转
  IF current_state != 'expected_state' THEN
    ROLLBACK; RAISE '状态冲突异常';
  END IF;

  UPDATE cf_examination SET exam_status = @newStatus WHERE id = @examId;
  COMMIT;
  ─────────────────────────────────────────────────────

  锁超时保护：
  SELECT ... FOR UPDATE WAIT 5;  -- 5秒等待超时，防止死锁
  -- 超时后抛出 LockWaitTimeoutException，触发重试
```

#### 8.3.3 DICOM接收高可用设计

**DicomIngestAgent 故障隔离**:

```
  DICOM设备推送（DICOM C-STORE）：
  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │   DICOM设备 ──▶ DICOM SCP (:11112)  ──▶ 内存缓冲区(队列)    │
  │                            │               │                 │
  │                            │         写队列 │ 读队列         │
  │                            ▼               ▼                 │
  │                   持久化缓冲区         Ingest线程池          │
  │                   (Redis/文件)           │                 │
  │                                     处理并写库              │
  └──────────────────────────────────────────────────────────────┘

  故障场景1：SCP进程崩溃
  保障：持久化缓冲区（Redis Stream）保留未处理消息
  恢复：SCP重启后，从缓冲区断点继续消费

  故障场景2：数据库写入阻塞
  保障：缓冲区设置最大深度（10000条），超限拒绝新接收并返回C-STORE-REFUSED
  监控：缓冲区深度 > 5000 触发P1告警

  故障场景3：存储服务不可用
  保障：内存缓冲区内最多1000个文件（每个100MB=100GB内存占用上限）
  超限：写入临时SSD缓冲目录，异步迁移至主存储
```

**健康检查与自动故障转移**:

```
  DicomIngestAgent 健康检查端点：
    GET /actuator/health/dicom-scp
    返回：{status: "UP", connectedDevices: 5, queueDepth: 150, lastReceived: "2026-03-25T10:00:00Z"}

  Nacos 注册中心心跳：
    心跳间隔：30秒
    心跳超时：3次（90秒无心跳则实例摘除）
    摘除后：负载均衡自动将流量切至其他实例

  DICOM设备侧：
    配置主备两个DicomIngestAgent地址
    主不可达时自动切换至备（设备端配置）
    超时时间：SCP响应超时 > 10s 设备自动重试
```

#### 8.3.4 存储灾备与RTO/RPO设计

**COLD层预热失败处理**:

```
  用户请求COLD层影像:
  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │  请求 ──▶ 检查存储层=hot? ──是──▶ 直接返回 (<100ms)          │
  │           │                                                │
  │           否                                                │
  │           ▼                                                │
  │  创建预热任务 (cf_storage_tier.status='preheating')         │
  │           │                                                │
  │           ▼                                                │
  │  后台复制至HOT层 ──成功──▶ 更新 cf_image.storage_tier='hot' │
  │           │                                                │
  │           失败(重试指数退避: 1min/5min/30min/2h)           │
  │           │                                                │
  │           ▼                                                │
  │  重试3次后 ──▶ 更新cf_storage_tier.status='failed'         │
  │            │                                                │
  │            ▼                                                │
  │  返回用户: 影像恢复中，请5分钟后重试（HTTP 202 Accepted）   │
  │  同时触发P2告警，人工核查                                    │
  └──────────────────────────────────────────────────────────────┘

  用户体验优化：
    前端轮询 /cloudfilm/image/{id}/preheat-status
    返回预热进度（0-100%）
    预热完成后WebSocket主动推送通知
```

**RTO/RPO指标**:

| 故障场景 | RTO（恢复时间目标） | RPO（恢复点目标） | 恢复策略 |
|---------|------------------|----------------|---------|
| HOT层单盘故障 | < 5分钟 | < 1分钟（RAID10） | 热备盘自动重构 |

#### 8.3.5 Temporal工作流边界场景处理

**检查点数据结构（Context Data）**:

```json
{
  "workflowId": "wf-20260325-001",
  "examId": 12345,
  "currentState": "qc_checking",
  "checkpoint": {
    "imageIds": [101, 102, 103],
    "qcScores": {"101": 92, "102": 85, "103": null},
    "processedImages": ["101", "102"],
    "pendingImages": ["103"],
    "lastAgent": "ImageQCAgent",
    "lastEventId": "msg-uuid-456"
  },
  "slaDeadline": "2026-03-25T14:00:00Z",
  "version": 7
}
```

**边界场景处理**:

```
  场景1：数据库回滚（检查点与DB不一致）
  ─────────────────────────────────────────────────────
  原因：WorkflowAgent更新了cf_examination状态，但数据库事务被回滚
  检测：每次状态变更后，验证cf_workflow_state.context_data.version 与
       cf_examination 表中的version字段一致
  处理：
    version不一致 → 从cf_workflow_state.context_data重建内存状态
                → 重新发布未处理的MQ事件（基于lastEventId）
                → 记录数据不一致日志，触发P1告警人工核查

  场景2：消息重放冲突
  ─────────────────────────────────────────────────────
  原因：Temporal重启后重放历史事件，但某些事件已在新workflow中处理
  检测：每个事件携带correlationId，检查cf_agent_event.message_id是否已处理
  处理：
    已处理 → 跳过该事件，记录"duplicate event skipped"
    未处理 → 正常处理
    冲突（同一imageId被两次QC评分）→ 以最新时间戳结果为准，写入cf_qc_issue历史表

  场景3：Agent扩缩容并发冲突
  ─────────────────────────────────────────────────────
  原因：缩容时Agent实例被Kill，正在处理的工作流任务中断
  处理：
    Temporal心跳超时（默认30s）→ 任务标记为TIMEOUT
    定时扫描cf_agent_task WHERE status='running' AND started_at < NOW()-10min
    → 重置为pending，分配给其他Agent实例重试
    → 重试次数+1，记录"agent-restart-recovery"
```

### 8.4 并发与性能设计

#### 8.4.1 DICOM接收吞吐带宽重评估

**原设计问题**: "100张/秒 × 500MB = 50GB/s"——此处"张"的定义模糊（应为帧，非文件）。

**重新评估**:

| 影像类型 | 单帧DICOM大小 | 单次检查帧数 | 单次检查总大小 | 100帧/秒所需带宽 |
|---------|------------|-----------|------------|--------------|
| DR胸片 | 8-15MB | 1-2帧 | 8-30MB | 800MB/s - 1.5GB/s |
| CT头颅 | 0.5-1MB | 30-50帧 | 30-50MB | 5GB/s - 10GB/s |
| CT胸部（薄层） | 0.5-1MB | 300-500帧 | 150-500MB | 30GB/s - 50GB/s |
| MRI | 0.3-0.8MB | 200-400帧 | 60-320MB | 6GB/s - 32GB/s |
| **实际设计目标** | — | **100帧/秒** | — | **约1-5GB/s（典型医院混合负载）** |

**带宽预算**:

```
  医院网络架构（建议）:
  DICOM设备区域交换机 (10GbE) ──▶ 核心交换机 (25GbE/40GbE) ──▶ 服务器
  ─────────────────────────────────────────────────────────────────
  峰值带宽需求（300床位三甲医院）:
  并发设备: 15台（DICOM设备）
  单设备峰值: 100MB/s
  汇聚带宽: 1.5GB/s
  预留冗余: ×1.5 = 2.25GB/s（25GbE网络可满足）

  若使用RDMA (RDMA over Converged Ethernet):
  ─────────────────────────────────────────────────────────────────
  单连接吞吐: 100-400MB/s（零拷贝，无CPU开销）
  15台设备 × 100MB/s = 1.5GB/s，可满足需求
  优势：CPU占用降低80%，延迟降低50%
```

**性能指标修正**:

| 指标 | 原设计 | 修正后 | 说明 |
|-----|-------|--------|------|
| DICOM接收吞吐 | 100张/秒 | **100帧/秒**（单检查内并发） | 以帧为处理单位，文件为存储单位 |
| 峰值处理能力 | — | 500MB/s（内网） | 25GbE网络下单个IngestAgent节点 |
| 接收成功率 | ≥99.95% | 保持 | 含网络重传保证 |

#### 8.4.4 数据库热点优化

```
  问题: cf_agent_event 高频写入(5万/天) + cf_image.qc_score 高频读写
  ───────────────────────────────────────────────────────────────
  优化方案1：表分区（cf_agent_event）
  ───────────────────────────────────────────────────────────────
  按月分区（PARTITION BY RANGE on created_at）：
    PARTITION p_2026_03: 2026-03-01 至 2026-03-31
    PARTITION p_2026_04: 2026-04-01 至 2026-04-30
    PARTITION p_future:   MAXVALUE

  写入热点消除：
    新数据只写入当月分区（无锁竞争）
    历史分区只读，查询走分区裁剪（PARTITION PRUNING）
    分区切换（SWITCH PARTITION）实现快速归档删除

  优化方案2：cf_image 行锁优化
  ───────────────────────────────────────────────────────────────
  qc_score 更新使用乐观锁替代悲观锁：
    UPDATE cf_image SET qc_score = @score WHERE id = @id;
    -- 无version字段时，每次更新都申请X锁

  改为非阻塞更新（延迟写）：
    ImageQCAgent 将评分结果写入Redis（key=image:qc:{id}）
    后台任务批量从Redis读出，批量UPDATE（减少行锁持有时间）
    批量大小：100条/次，间隔：5秒

  优化方案3：cf_audit_log 分离写
  ───────────────────────────────────────────────────────────────
  审计日志异步写入（不参与主事务）：
    Spring @Async + 线程池隔离
    主线程：记录AuditEvent至ThreadLocal
    异步线程：批量聚合并发（100条/次，100ms超时）
    写入失败：降级至本地文件队列，重试保证不丢失
```

**鉴权规则**: 无JWT要求，使用访问码（6位数字）验证，支持外部患者访问

**安全措施**:
```
访问码暴力破解防护:
  └── 单IP 5次错误后锁定15分钟
  └── 验证码错误日志记录（SecurityAuditAgent）

分享链接有效期:
  └── 默认7天（cf_image_share.expire_time）
  └── 超期自动失效，返回410 Gone

  └── ShareAgent在响应前自动检查脱敏状态
  └── 未脱敏影像拒绝对外返回
```

```

     ▼

     │
     ▼
  目标服务处理
     │
     ▼
  [6] ResponseWrapFilter    统一响应格式（非必须，按需启用）

## 13. 安全架构设计

### 13.1 JWT安全加固方案

#### 13.1.1 现状与风险

当前实现存在以下风险：

- JWT密钥硬编码于 `application.yml`，泄露后攻击者可伪造任意Token
- HS256对称算法要求所有节点存储密钥，增加泄露面
- 无Token黑名单机制，泄露后无法吊销

#### 13.1.2 升级方案：RS256 + 密钥轮换

```
  密钥体系设计:
  ┌──────────────────────────────────────────────────────────────┐
  │  签名密钥对 (RS256 非对称加密):                               │
  │  ├── 私钥 (Private Key): 用于签名，仅存于签名服务器/KMS       │
  │  ├── 公钥 (Public Key): 用于验签，可分发至所有节点             │
  │                                                              │
  │  Access Token:                                               │
  │  ├── Payload: userId, roles, permissions, exp, iat           │
  │  ├── 签名: 使用私钥RS256签名                                 │
  │  └── 有效期: 120分钟（短令牌）                                │
  │                                                              │
  │  Refresh Token:                                              │
  │  ├── Payload: userId, tokenVersion(递增), exp                 │
  │  ├── 签名: 使用私钥RS256签名                                 │
  │  ├── 有效期: 7天                                              │
  │  └── 存储: Redis (tokenVersion作版本号，支持强制失效）        │
  └──────────────────────────────────────────────────────────────┘

  Token吊销机制（Redis黑名单）:
  ───────────────────────────────────────────────────────────────

    → Redis HSET token_blacklist {jti} {expireAt}

### 13.4 访问码安全加固

#### 13.4.1 当前风险

`IdUtils.java` 使用 `Math.random()` 生成访问码——线性同余算法可被预测，医疗影像分享存在严重安全隐患。

#### 13.4.2 修复方案

```java
// 修复后 - 使用 SecureRandom（密码学安全随机数生成器）
public static String accessCode() {
    SecureRandom random = new SecureRandom();
    int code = random.nextInt(1_000_000);  // 0 ~ 999999
    return String.format("%06d", code);
}

// 增强方案：动态验证码（非固定6位数字）
public static String generateAccessCode(AccessCodeType type) {
    SecureRandom random = new SecureRandom();
    switch (type) {
        case NUMERIC_6:
            return String.format("%06d", random.nextInt(1_000_000));
        case ALPHANUMERIC_8:  // 更强安全场景
            String chars = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789"; // 去混淆字符
            StringBuilder sb = new StringBuilder(8);
            byte[] bytes = random.generateSeed(8);
            for (int i = 0; i < 8; i++) {
                sb.append(chars[Math.abs(bytes[i] % chars.length())]);
            }
            return sb.toString();
        default:
            throw new IllegalArgumentException("Unknown type");
    }
}
```

#### 13.4.3 暴力破解防护实现

```java
// ShareAccessRateLimiter - Redis滑动窗口限流
@Service
@RequiredArgsConstructor
public class ShareAccessRateLimiter {
    private final StringRedisTemplate redisTemplate;

    private static final int MAX_FAILURES = 5;    // 5次错误
    private static final int LOCKOUT_SECONDS = 900; // 锁定15分钟

    public void recordFailure(String shareNo, String ipAddress) {
        String key = "share:fail:" + shareNo + ":" + ipAddress;
        Long failures = redisTemplate.opsForValue().increment(key);
        if (failures != null && failures == 1) {
            redisTemplate.expire(key, LOCKOUT_SECONDS, TimeUnit.SECONDS);
        }
        if (failures != null && failures >= MAX_FAILURES) {
            redisTemplate.opsForValue().set(
                "share:lockout:" + shareNo + ":" + ipAddress, "1",
                LOCKOUT_SECONDS, TimeUnit.SECONDS
            );
            // 触发安全告警（AUD-006）
            securityAuditAgent.recordAnomaly("AUD-006", ipAddress,
                "访问码暴力破解嫌疑: shareNo=" + shareNo + ", failures=" + failures);
        }
    }

    public boolean isLockedOut(String shareNo, String ipAddress) {
        return Boolean.TRUE.equals(
            redisTemplate.hasKey("share:lockout:" + shareNo + ":" + ipAddress));
    }

    public void recordSuccess(String shareNo, String ipAddress) {
        redisTemplate.delete("share:fail:" + shareNo + ":" + ipAddress);
    }
}
```

### 14.3 断点续传记录表

```sql
CREATE TABLE cf_upload_chunks (
    id                BIGINT          NOT NULL PRIMARY KEY,

### 14.4 分片上传流程

```
  Phase 1: 初始化上传
  ───────────────────────────────────────────────────────────────
  POST /cloudfilm/image/upload/init
  Body: { fileName, fileSize, fileHash(MD5), examId }
  ───────────────────────────────────────────────────────────────
  后台处理:
    1. 检查upload_id是否已存在（断点续传场景）
       → 存在：返回已上传分片列表（前端跳过已上传分片）
    2. 生成 upload_id（UUID）存入Redis（TTL=24h）
    3. 计算总分片数 = ceil(fileSize / 10MB)
    4. 创建 cf_upload_chunks 记录（全部分片，status='uploading'）
    5. 返回 { uploadId, totalChunks, chunkSize: 10485760 }

  Phase 2: 并发上传分片
  ───────────────────────────────────────────────────────────────
  POST /cloudfilm/image/upload/chunk
  Body: multipart/form-data
    - uploadId: upload_id
    - chunkIndex: 0, 1, 2, ...
    - chunkHash: 分片MD5
    - file: <binary>
  ───────────────────────────────────────────────────────────────
  后台处理:
    1. 校验 uploadId 是否存在且未过期（Redis）
    2. 校验分片索引范围（0 ≤ index < totalChunks）
    3. 计算接收分片MD5，与 chunkHash 比对
       → 不一致：返回400 BadRequest，客户端重试该分片
    4. 写入临时目录: /uploads/temp/{upload_id}/{chunkIndex}.dcm
    5. 更新 cf_upload_chunks SET status='completed'
    6. 返回 { chunkIndex, status: 'completed' }

  Phase 3: 合并分片
  ───────────────────────────────────────────────────────────────
  POST /cloudfilm/image/upload/merge
  Body: { uploadId, fileHash }
  ───────────────────────────────────────────────────────────────
  后台处理:
    1. 校验所有分片 status='completed'
       → 有遗漏：返回 400，列出未上传分片索引
    2. 顺序读取分片，追加写入 /uploads/images/{date}/{uuid}.dcm
    3. 计算合并后文件MD5，与 fileHash 比对
       → 不一致：删除合并文件，标记失败，要求重新上传
    4. DicomParser 解析 → Saga Step 1-4 写库（见8.3.2节）
    5. 更新 cf_upload_chunks SET status='merged'
    6. 删除所有临时分片文件
    7. 返回 { imageId, status: 'received' }

  超时清理（每日凌晨03:00 Cron）:
  ───────────────────────────────────────────────────────────────
  SELECT upload_id FROM cf_upload_chunks
  WHERE status='uploading' AND updated_at < NOW()-24h;
  → 标记为failed，删除临时文件，记录cf_audit_log

  前端断点续传:
  ───────────────────────────────────────────────────────────────
  1. 上传前调用 /cloudfilm/image/upload/init（带fileHash）
  2. 后台返回 { uploadId, completedChunks: [0,1,2] }
  3. 前端跳过已完成分片，仅上传剩余分片
```

    → 直接进入人工阅片流程
    → WorkflowAgent记录SLA不受影响

  Level 3: 紧急降级
    全部GPU不可用 → 自动扩展CPU推理实例（量化INT8模型）
    → 显存减少50%，速度降低30%
    → SLA自动延长50%

  降级恢复:
  ─────────────────────────────────────────────────────────────
  每30秒探测AI推理服务可用性
  连续3次成功 → Circuit Breaker半开 → 逐步恢复流量
```

### 16.2 P0级事件响应（核心业务中断 / 数据泄露）

#### 16.2.1 事件识别（0-5分钟）

```
  触发条件（任一满足即进入P0响应）:
  ├── SystemMonitorAgent触发P0告警（DICOM全断/数据库不可用）
  ├── SecurityAuditAgent触发AUD-003/AUD-007告警（越界/篡改）
  ├── 内部人员报告异常（患者投诉数据泄露）
  └── 外部通报（监管机构/合作医院通知）

  初步确认清单:
  [ ] 确认事件真实性（排除误报）
  [ ] 记录发现时间、报告人、初始描述
  [ ] 创建事件工单（标题：[P0安全事件] + 简要描述）
  [ ] 通知安全运营工程师 + 系统管理员 + 数据库管理员
  [ ] 记录进入响应状态的时间
```

#### 16.2.2 止血操作（5-15分钟）

```
  数据泄露止血:
  ───────────────────────────────────────────────────────────────
  步骤1：确认泄露范围
    → 查询cf_audit_log中AUD-003/AUD-009记录
    → 确定数据类型（影像/报告/患者信息/诊断内容）
    → 确定受影响患者数量（DISTINCT patient_id）
    → 确定泄露时间窗口

  步骤2：阻断继续泄露
    账号泄露 → 禁用账号（sys_user.status='1'）
    API泄露 → 撤销API Key，强制刷新所有Token
    分享链接泄露 → 删除所有相关分享记录
    内部人员泄露 → 隔离所有系统权限

  步骤3：保全证据
    → 备份cf_audit_log（完整审计记录）
    → 备份相关数据库表
    → 截图所有可疑操作日志
    → 通知法务/合规官

  系统中断止血:
  ───────────────────────────────────────────────────────────────
  DICOM服务中断：
    1. 检查DicomIngestAgent健康端点
    2. SCP进程崩溃 → 重启进程，验证队列积压量
    3. 数据库写入阻塞 → 检查连接池（druid连接数）
    4. 存储不可用 → 切换至备用存储路径
    5. 通知DICOM设备切换至备用节点

  数据库不可用：
    1. 确认连接问题还是实例故障
    2. 切换至只读副本（如果为主库故障）
    3. 应用层降级：标记所有写操作为"pending_retry"
    4. 恢复后重放pending队列
```

#### 16.2.3 根因分析与恢复（15-60分钟）

  根因分析模板:
  ┌──────────────────────────────────────────────────────────────┐
