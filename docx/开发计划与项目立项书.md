## 一、项目立项书

### 1.1 项目概述

| 项目要素 | 说明 |
|----------|------|
| **项目名称** | D-Site V1.0 云胶片管理系统 |
| **项目性质** | 医疗信息化 SaaS 产品（B/S 架构），支持私有化部署 |
| **立项背景** | 传统物理胶片成本高、携带不便、难以共享；医疗机构影像数据孤岛严重；患者无法便捷获取自身检查资料 |
| **核心问题** | 为医疗机构提供 DICOM 影像云端化管理，打通"采集-诊断-报告-分享"全流程闭环 |
| **目标用户** | 管理员（系统运维）、放射科医生（阅片/报告）、临床医生（申请/查看）、患者（查看/分享） |

### 1.2 产品定位

D-Site 云胶片管理系统是一套基于 Web 的医学影像管理平台，为医疗机构提供医学影像存储、查看、诊断报告生成及分享的一体化解决方案。系统采用 B/S 架构，支持医学影像的云端化管理和跨平台访问，定位为医疗机构内部影像管理和对外患者服务的统一入口。

### 1.3 商业价值

| 价值维度 | 说明 |
|----------|------|
| **降低成本** | 替代物理胶片，节省材料采购与物流分发成本 |
| **提升效率** | 在线阅片、AI 解读预填，缩短报告周期，减少医生重复劳动 |
| **改善体验** | 患者自助查询影像和报告，扫码即分享给其他医院，无需携带实体胶片 |
| **合规保障** | 全链路操作审计日志，满足国家卫健委《医疗数据安全管理办法》要求 |
| **扩展空间** | Agent 架构支持后续 AI 辅助诊断、多院区医联体联动 |

### 1.4 产品目标（V1.0 MVP）

**核心验收标准**：患者登记 → 开检查单 → 上传 DICOM 影像 → 撰写诊断报告 → PDF 导出 → 生成分享链接 → 患者扫码查看，**全流程端到端跑通，无阻断性 Bug**。

**性能基线**：

| 指标 | 目标值 |
|------|--------|
| 系统响应时间 | < 3 秒 |
| 影像首帧加载时间 | < 5 秒 |
| 并发用户数 | ≥ 100 |
| 系统可用性 | ≥ 99.5% |
| 单文件上传上限 | 500 MB |

### 2.4 核心流程端到端验收

Sprint 2 完成后，执行以下端到端测试：

```
放射科医生操作流程：
1. 登录系统 → 进入患者管理 → 新增患者"张三"
2. 进入检查管理 → 为张三新建检查单（检查类型：胸部 CT，检查原因：咳嗽）
3. 检查单状态：pending（待检查）
4. 进入影像管理 → 上传张三的 DICOM 文件 → 缩略图自动生成
5. 切换至在线阅片 → OHIF Viewer 加载影像 → 窗宽窗位调整 → 标记关键影像
6. 检查单状态更新：pending → in_progress → completed
7. 进入报告管理 → 新增报告（临床资料：咳嗽一周...；检查所见：右肺...；诊断意见：...）
8. 点击"导出 PDF" → 下载诊断报告 PDF（含影像截图和中文字体）
9. 检查单状态更新：completed → reported
10. 进入分享管理 → 创建分享（有效期 7 天，允许下载）→ 获得分享链接 + 提取码

患者操作流程：
11. 打开分享链接 → 输入 6 位提取码 → 验证通过
12. 查看影像列表 → 点击"查看影像" → H5 影像查看页（简化 OHIF）
```

## 四、技术设计要点

### 4.1 数据库事务边界设计

影像上传涉及文件存储（外部 IO）与数据库写入（事务操作），当前代码两者在同事务中，存在文件上传成功但数据库回滚无法撤销文件的风险。**MVP 阶段采用简化 Saga 模式**：

```
Step 1: 事务内创建 cf_image 记录（status='pending_upload'）
Step 2: 事务外调用 storageService.upload() 写文件
        失败 → UPDATE status='upload_failed'，抛异常
Step 3: UPDATE cf_image SET status='received', image_path=@path WHERE id=@id AND status='pending_upload'
        （条件更新，防止重复处理）
```

### 4.2 存储服务注入约定

`StorageService` 有两个实现：`LocalStorageService`（默认）和 `MinioStorageService`。当前注入方式为：

```java
@Autowired
@Qualifier("localStorageService")
private StorageService storageService;
```

如需切换至 MinIO：
1. 将 `application.yml` 中 `minio.enabled=true`
2. 将 `@Qualifier("localStorageService")` 改为 `@Qualifier("minioStorageService")`

### 4.3 DicomParser 已知 Bug

**位置**: `src/medical-system/src/main/java/com/dsite/medical/cloudfilm/dicom/DicomParser.java:63`

**问题**: 第 63 行 `info.setStudyTime(getString(attrs, Tag.StudyDate))` — 误用 `Tag.StudyDate` 读取 StudyTime 字段，应改为 `Tag.StudyTime`。

**影响**: 上传的 DICOM 检查时间会错误地写入 StudyDate 日期而非 StudyTime 时间。**Sprint 2 开始前必须修复。**

### 4.4 访问码安全修复

当前 `IdUtils.accessCode()` 推断使用 `Math.random()` 生成 6 位提取码，需改为密码学安全的随机数生成器：

```java
import java.security.SecureRandom;

public static String accessCode() {
    SecureRandom random = new SecureRandom();
    int code = random.nextInt(1_000_000);  // 0 ~ 999999
    return String.format("%06d", code);
}
```

## 六、V2.0 Agent 架构演进路线（长期规划）

基于 PACS-Agent 架构设计文档（`docx/PACS-Agent架构设计文档.md`），分阶段演进：

### Phase 1: 当前 V1.0 — 单体直接调用

现有代码结构，Controller 直接调用 Service，无 MQ 解耦。

### Phase 2: 引入消息队列（预计 V1.x，约 4 周）

**目标**: Agent 间事件解耦，WorkflowOrchestrationAgent 管理检查全流程状态机。

- 引入 RabbitMQ + Spring AMQP
- 定义核心 Topic：`pacs.dicom.received`、`pacs.qc.passed`、`pacs.qc.failed`、`pacs.ai.completed`、`pacs.report.ready`
- WorkflowOrchestrationAgent：检查状态机 + SLA 超时监控 + 异常补偿
- StorageSchedulerAgent：HOT→WARM→COLD 分级存储迁移（MinIO 激活）
- ImageDeliveryAgent：WADO-RS 完整实现 + 影像预加载策略
- **新建数据库表**: `cf_workflow_state`、`cf_agent_event`、`cf_agent_task`

### Phase 3: AI 诊断集成（预计 V2.0，约 6 周）

**目标**: 接入 AI 推理能力，实现智能诊断辅助。

- AIDiagnosisAgent：模态检测 → 模型路由（CT/MRI/DR）；对接 ModelScope API；多模型融合；置信度量化
- ReportGenerationAgent 升级：AI 解读结果自动填充"检查所见"段落；历史检查对比分析
- SecurityAuditAgent 基础版：影像访问全链路日志；批量下载异常检测（AUD-001 规则）；DICOM 对外分享自动脱敏
- **新建数据库表**: `cf_qc_issue`、`cf_audit_log`

### Phase 4: 完整 Agent 集群（预计 V2.x，约 8-12 周）

**目标**: 生产级高可用，支持弹性扩缩容。

- Temporal.io 工作流引擎：持久化工作流（系统重启不丢状态）；自动重试与补偿
- K8s 容器化：各 Agent 独立 Deployment；HPA 自动扩缩容（队列积压>100 时自动扩容）
- Kafka 替代 RabbitMQ：事件溯源；高吞吐 DICOM 流处理
- 全链路可观测性：Micrometer + Prometheus + Grafana 大屏；Zipkin 分布式追踪
- **新建数据库表**: `cf_storage_tier`
