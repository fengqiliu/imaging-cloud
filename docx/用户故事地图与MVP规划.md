# D-Site V1.0 用户故事地图与MVP规划

## 1. 用户故事地图

### 1.1 用户角色与核心故事

#### 管理员 (Admin)
| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-01 | 作为管理员，我可以管理系统用户（增删改查），以便管理账号访问权限 | P0 |
| US-02 | 作为管理员，我可以管理用户角色和权限，以便控制不同角色的系统访问范围 | P0 |
| US-03 | 作为管理员，我可以管理院区信息，以便支持多院区运营 | P0 |
| US-04 | 作为管理员，我可以配置系统菜单和按钮，以便定制系统功能入口 | P0 |
| US-05 | 作为管理员，我可以管理系统字典（类型和数据），以便维护业务配置数据 | P0 |
| US-06 | 作为管理员，我可以查看操作日志，以便审计用户行为 | P0 |
| US-07 | 作为管理员，我可以管理系统参数，以便调整系统运行配置 | P0 |
| US-08 | 作为管理员，我可以重置用户密码，以便帮助用户恢复账号 | P1 |

#### 放射科医生 (Radiologist)
| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-09 | 作为放射科医生，我可以上传DICOM影像文件，以便将检查影像存入系统 | P0 |
| US-10 | 作为放射科医生，我可以在线浏览医学影像，以便进行诊断分析 | P0 |
| US-11 | 作为放射科医生，我可以查看检查列表和详情，以便了解待处理任务 | P0 |
| US-12 | 作为放射科医生，我可以撰写诊断报告，以便记录诊断结果 | P0 |
| US-13 | 作为放射科医生，我可以导出PDF报告，以便提供纸质诊断依据 | P0 |
| US-14 | 作为放射科医生，我可以修改诊断报告，以便更正诊断内容 | P0 |
| US-15 | 作为放射科医生，我可以管理检查状态（待检查→检查中→已完成→已出报告），以便跟踪检查进度 | P0 |
| US-16 | 作为放射科医生，我可以标记关键影像，以便后续快速定位重点图像 | P1 |

#### 临床医生 (Clinician)
| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-17 | 作为临床医生，我可以搜索和查看患者信息，以便了解患者基本情况 | P0 |
| US-18 | 作为临床医生，我可以为患者创建检查申请单，以便安排影像检查 | P0 |
| US-19 | 作为临床医生，我可以查看患者的历史检查记录，以便追踪病情变化 | P0 |
| US-20 | 作为临床医生，我可以查看患者的影像和报告，以便辅助临床决策 | P0 |
| US-21 | 作为临床医生，我可以编辑检查申请单，以便修改检查信息 | P0 |

#### 患者 (Patient)
| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-22 | 作为患者，我可以查看自己的检查报告，以便了解检查结果 | P0 |
| US-23 | 作为患者，我可以查看自己的医学影像，以便了解自身健康状况 | P0 |
| US-24 | 作为患者，我可以创建影像分享链接，以便将影像分享给他人 | P0 |
| US-25 | 作为患者，我可以设置分享有效期和提取码，以便控制分享安全 | P0 |
| US-26 | 作为患者，我可以取消已创建的分享链接，以便终止分享 | P0 |
| US-27 | 作为患者，我可以通过链接和提取码查看分享的影像，以便获取检查资料 | P0 |

## 4. 技术架构说明

### 4.1 技术选型

| 层级 | 技术栈 |
|------|--------|
| 前端 | Vue 2 + Element UI + OHIF Viewer |
| 后端 | Java 21 + Spring Boot + Spring Security |
| 数据访问 | MyBatis Plus |
| 认证授权 | JWT |
| 数据库 | SQL Server 2019 |
| DICOM处理 | dcm4che |
| 存储 | MinIO / 本地存储 |
| PDF生成 | iText / OpenPDF |

### 4.2 模块依赖

```
medical-admin (应用入口)
  └── medical-framework
        ├── medical-common (通用工具)
        ├── medical-system (系统管理)
        ├── medical-dicom (DICOM处理)
        ├── medical-storage (文件存储)
        ├── medical-image (图像处理)
        ├── medical-report (报告生成)
        ├── medical-share (分享功能)
        └── medical-ai (AI功能 - MVP排除)
```

## 5. 验收标准

### 5.1 Sprint 1 验收标准

- [ ] 用户可以登录系统
- [ ] 管理员可以管理用户账号
- [ ] 管理员可以管理角色权限
- [ ] 管理员可以管理院区
- [ ] 管理员可以查看操作日志
- [ ] 可以对患者进行CRUD操作
- [ ] 可以对检查单进行CRUD操作

### 5.2 Sprint 2 验收标准

- [ ] 可以上传DICOM文件并解析
- [ ] 可以在浏览器中查看医学影像
- [ ] 自动生成影像缩略图
- [ ] 可以撰写和编辑诊断报告
- [ ] 可以导出PDF格式报告
- [ ] 可以创建带提取码的分享链接
- [ ] 外部用户可以通过链接查看影像

### 5.3 MVP整体验收标准

1. 核心业务流程跑通: 患者→检查→影像→报告→分享
2. 系统响应时间 < 3秒
3. 影像加载时间 < 5秒
4. 无阻断性Bug

## 6. 风险与依赖

### 6.1 关键风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| OHIF集成复杂度 | 高 | 提前调研，准备备选方案 (Cornerstone.js) |
| DICOM解析兼容性 | 中 | 测试多种DICOM文件格式 |
| PDF生成性能 | 中 | 优化模板，预览与导出分离 |
| 项目时间紧 | 中 | 优先保障核心功能，非核心延后 |

### 6.2 外部依赖

| 依赖 | 说明 | 负责人 |
|------|------|--------|
